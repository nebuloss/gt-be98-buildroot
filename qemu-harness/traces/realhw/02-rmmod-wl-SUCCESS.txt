## STEP 2 — rmmod wl  => SUCCESS, SSH SURVIVED  [RE-CONFIRMED]
- `rmmod wl` exit=0; wl GONE; SSH stayed up (ran over wired br0).
- br0 dropped ONLY wl0-3; eth0-3 remained members (lifeline intact, as designed).
- All shared deps stayed loaded (hnd,cfg80211,emf,igs,wlshared,bcm_pcie_hcd,
  bdmf,rdpa_gpl,bcm_knvram) — available for dhd.
- All 4 PCIe endpoints (0000-0003:01:00.0) became driverless (driver= empty)
  => chips freed and available for dhd to claim. [RE-CONFIRMED]
- rmmod dmesg: clean wl teardown (bsscfg free, M2MDMA deinit, pktfwd fini).
  => CONFIRMS the SPEC/ROADMAP claim that wl and dhd compete for the same
     class-0x028000 device and rmmod wl cleanly releases them.

## STEP 3 — insmod dhd (high debug) => DEVICE WEDGED / DROPPED OFF NET
- cmd: insmod dhd.ko iface_name=wl dhd_console_ms=250 dhd_msg_level=0x617e7fff
- insmod did NOT return; SSH session hung; device then "No route to host".
- Interpretation: dhd probe of the REAL 6726b0 silicon hung/faulted the kernel
  before dmesg could be drained over SSH. Watchdog auto-reset to committed v34
  expected. (dmesg from the dhd probe was NOT recoverable over the live session
  because the network path died with the hang.)
- This itself is a FINDING: loading the stock closed dhd.ko against the live
  chips — with the rest of the merlin wl/runner stack still resident — wedges
  the box rather than cleanly probing. See M3 verdict + console-capture caveat.
